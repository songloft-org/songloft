## 简介

MiMusic 是一个基于 Go 和 Chi 框架的轻量级音乐服务器，专门设计用于提供高效的音乐缓存系统。该系统通过智能缓存机制、并发下载管理和预加载策略，为用户提供流畅的音乐播放体验。

音乐缓存系统的核心目标是：
- **统一缓存接口**：从哈希系统迁移到歌曲ID系统，提供统一的缓存访问方式
- **专注的缓存管理**：缓存处理器专注于缓存统计、清理和配置管理
- **统一播放端点**：播放功能集中在 SongHandler，提供 `/api/v1/songs/{id}/play` 统一播放接口
- **降低延迟**：通过本地缓存减少网络请求次数
- **提高可靠性**：在网络不稳定时提供离线播放能力
- **优化带宽**：避免重复下载相同内容
- **增强用户体验**：支持无缝播放和快速切换
- **智能内存管理**：通过 LRU 算法优化缓存空间使用
- **完整的缓存管理**：提供统计、清理和配置管理功能
- **多键缓存支持**：基于歌曲ID的多键缓存系统，防止缓存冲突
- **智能冲突解决**：确保不同来源的同一首歌曲不会相互干扰
- **精确缓存隔离**：插件来源和外链来源的缓存完全隔离

## 性能考虑

### 缓存策略优化

1. **智能目录分层**：使用歌曲ID的三级目录结构，避免单个目录中文件过多
2. **并发控制**：通过 `inflight` 映射避免重复下载相同内容
3. **内存管理**：使用 `ServeFile` 直接返回文件，避免将整个文件加载到内存
4. **超时控制**：设置合理的 HTTP 超时时间，防止长时间阻塞
5. **LRU淘汰**：基于访问时间的智能缓存淘汰，优化存储空间使用
6. **缓存控制头**：使用适当的 HTTP 头信息优化浏览器缓存行为
7. **多键缓存**：基于歌曲ID的多键缓存系统，防止缓存冲突
8. **智能冲突解决**：确保不同来源的同一首歌曲不会相互干扰
9. **精确缓存隔离**：插件来源和外链来源的缓存完全隔离

### 存储优化

1. **文件系统优化**：SQLite 使用 WAL 模式提高并发性能
2. **连接池管理**：合理设置数据库连接池大小
3. **磁盘空间管理**：定期清理无效缓存文件
4. **索引优化**：`cache_hash` 唯一索引确保查询性能
5. **LRU索引优化**：内存中的 LRU 索引提高访问速度
6. **缓存键优化**：智能的缓存键生成算法提高缓存命中率

### 网络优化

1. **重定向处理**：使用 HEAD 请求解析重定向，减少不必要的数据传输
2. **内容类型验证**：确保下载的是有效的音频文件
3. **范围请求支持**：利用 `ServeFile` 自动支持 Range 请求
4. **预加载优化**：后台下载不阻塞主线程
5. **HTTP方法优化**：HEAD 请求专门用于状态检查，避免不必要的数据传输
6. **缓存键冲突避免**：防止跨数据库重建导致的缓存冲突

### 歌曲ID系统优化

1. **去重效率**：通过 `song_id` 字段实现高效的重复检测
2. **原子操作**：UPSERT 操作确保数据一致性
3. **条件更新**：避免覆盖已有的正确元数据
4. **索引优化**：唯一索引确保查询性能
5. **多键缓存**：支持插件来源和外链来源的精确缓存隔离

### LRU算法优化

1. **双索引机制**：内存索引 + 文件 mtime 双重保证
2. **读写分离**：LRU索引使用 RWMutex 实现读写分离
3. **批量淘汰**：按访问时间排序后批量删除
4. **实时监控**：配置更新后立即执行缓存淘汰
5. **智能堆管理**：使用 container/heap 实现最大堆，优化内存使用

### 缓存管理API优化

1. **批量操作**：清理缓存时一次性删除所有文件
2. **异步处理**：配置更新后异步触发淘汰检查
3. **状态检查**：HEAD 请求专门用于检查下载状态
4. **错误处理**：完善的错误处理和日志记录
5. **多键缓存支持**：支持基于歌曲ID的多键缓存管理

### 播放端点优化

1. **统一接口**：所有歌曲类型通过同一端点播放，简化客户端集成
2. **类型分发**：根据歌曲类型自动选择最优的播放策略
3. **缓存策略**：本地歌曲直接缓存，远程歌曲智能缓存，电台歌曲不缓存
4. **错误处理**：插件来源歌曲失败时自动切换音源
5. **多键缓存集成**：与多键缓存系统无缝集成
